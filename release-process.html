The Release Process - Trading Functionality for Releasability

The Headaches of Releasing

As web developers it can be easy to lose sight of the end goal of the code we write. We can get caught up in new technologies or focus on nagging edge cases that we know exist since we built the system. But at the end of the day the goal is to release new code to the customer and do so in a timely fashion. Unfortunately, releasing code is not always a straight forward process. Often times cooridination is needed with other products and this can cause tedious extra (and arguably unneccessary) work for developers. No developer wants to spend time looking through old commits and seeing what can and can't be released, manage multiple branches with different code, or adding in logic to hide features that aren't ready to be released. In my experience, making sacrifices around functionality to expedite and simplify the release process can ultimately benefit any product's development cycle.

Leveraging Self-Sufficient Software

What often hampers or delays a product's ability to release is sharing a codebase with one or mulitple other products. From a functionality perspective this can have obvious benefits; a product's code can directly accss other products' datasources, there is no needed communication between separate applications, and there are fewer applications that need managed, to name a few. Although this seems efficient, it can make it a disaster to release. Any time that one product in the codebase wants to release the other products need to ensure all of its code is production ready. And although in theory many development teams try to ensure all code merged into the main branch is "releasable", in reality that often is much more easier said than done. As the number of products that use the codebase grows, you have to rely on more development teams to adhere by that mantra.

The solution is to create separate codebases and applications for the underlying logic that is needed by your product. Create contracts with other products whose data yours relies on and accept the extra latency from calling those applications. For front end applications, it may mean accepting a small "blip" when navigating from one product to another in the user facing web application. But by gaining the indpendence to release on your terms and schedule new code and features will make its way to customers faster. Other products will not need to care if and when you release, and vice versa. Ultimately it will create a quicker and more efficient development cycle, and as a developer allows you to focus on what you like doing best; writing code.
